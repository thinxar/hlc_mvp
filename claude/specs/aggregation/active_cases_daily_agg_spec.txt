Aggregation Spec - Active Cases, Daily Branch Activity
======================================================

Source: generated/all_cases.jsonl (record schema: specs/data_gen/data_generation.txt)

Companion to: specs/aggregation/monthly_branchwise_agg_spec.txt
Same source, same status mapping, same sparse-output convention. The
monthly report covers historical volume; this report covers the
"currently active" working queue - cases that are recent-and-pending
or have recent document actions.


Parameters
----------
Single-source-of-truth knobs referenced throughout this spec. Change
the value here and every derived bound below (case selection arm B,
cal_date enumeration, Rules 1 and 8) follows automatically.

  WINDOW_DAYS = 14     # length of the reporting / "recently actioned" window
                        # in calendar days, inclusive. Window spans
                        # [REF_DATE - (WINDOW_DAYS - 1), REF_DATE].

All other numeric thresholds in this spec (e.g. the 90-day arm-A
lookback) are deliberately NOT parameterised - if they need to vary,
add a named parameter here first.


Report - Active Cases, Daily Branch Activity
--------------------------------------------
Grain:         one row per (cal_date, branchCode). cal_date is a DAILY
               SNAPSHOT DATE that iterates across the reporting window
               (see "Reporting window" below); it is NOT the source
               record's requestDate. A single source document may be
               counted in pendingDocuments across many cal_dates (every
               day from its uploadedOn until the day before its
               actionOn), in submittedDocuments on exactly one cal_date
               (its uploadedOn), and in processedDocuments on exactly
               one cal_date (its actionOn).
Output format: flat JSON object per row (file-level container - JSON array
               or NDJSON - decided when the reporting module is wired up).
Output file:   generated/active_cases_branchwise.json

Top-level fields
----------------

Field              | Type    | Format / Constraint                                        | Example
-------------------|---------|------------------------------------------------------------|------------
id                 | integer | unique running sequence within this report, 1..N, assigned | 1
                   |         | AFTER the final sort (rule 8) so id order matches row order|
branchCode         | string  | 3 or 4 chars; grouping key                                 | "359"
branchName         | string  | from source record                                         | "SEHORE"
divisionName       | string  | from source record                                         | "Bhopal"
doCode             | string  | from source record; constant per divisionName              | "201"
Zone               | string  | from source record (key preserved as "Zone", not camel)    | "CZ"
cal_date           | string  | YYYY-MM-DD; a day in the reporting window (see below)      | "2026-04-17"
pendingDocuments   | integer | open backlog AS OF cal_date (see rule 3)                   | 2
submittedDocuments | integer | documents uploaded ON cal_date (see rule 4)                | 41
processedDocuments | integer | documents actioned ON cal_date (see rule 5)                | 45
perApprover        | array   | breakdown of processedDocuments by approver; see schema    | see below

perApprover element schema
--------------------------

Field              | Type    | Format / Constraint                                        | Example
-------------------|---------|------------------------------------------------------------|------------
approvedBy         | string  | officer id from the source document's approvedBy (8-digit)| "57988792"
accepted           | integer | documents with status "accepted" actioned by this approver| 15
rejected           | integer | documents with status "rejected" actioned by this approver| 2

Invariants:
  per-element:  accepted + rejected >= 1 (every emitted element is non-empty)
  per-bucket:   SUM(perApprover[].accepted + perApprover[].rejected)
                    == top-level processedDocuments
Only documents with status in {"accepted","rejected"} contribute to
perApprover. Pending documents have no approver and are excluded.
perApprover is sorted by approvedBy ASC and contains no duplicate
approvedBy values (one entry per approver per bucket). An approver
whose contribution to this bucket is purely accepted (or purely
rejected) simply has the other count at 0 - they are NOT emitted as
two separate elements.

Case selection
--------------
Let REF_DATE be the run's reference date - resolved from environment
variable REPORT_TODAY (YYYY-MM-DD) if set, otherwise the local system
date at run time. Dates in the source are date-only strings; compare
as ISO strings, no timezone math required.

A source case qualifies for this report when EITHER condition holds
(logical OR; a case meeting both is included once):

  A. Recent-and-pending:
     requestDate >= REF_DATE - 89 days (i.e. within the last 90 days),
     AND the case has at least one document with status = "pending".

  B. Recently actioned:
     the case has at least one document whose actionOn falls within
     the reporting window [REF_DATE - (WINDOW_DAYS - 1), REF_DATE]
     (i.e. within the last WINDOW_DAYS days). Only documents with
     status in {"accepted","rejected"} have a non-empty actionOn,
     so this arm implicitly considers only those.

Field-name mapping: the user-facing term "approvedOn" maps to the
source field actionOn (documented in specs/data_gen/data_generation.txt).
There is no separate approvedOn on documents - actionOn is the date
the document was actioned (accepted or rejected), which is what this
report treats as the approval/rejection timestamp.

Once a case qualifies, ALL of its documents are eligible to contribute
to bucket counts (no case-level document filter); the per-cal_date
rules 3, 4 and 5 decide on a per-document, per-day basis whether the
document is counted in that day's pending, submitted, or processed
total.

Reporting window (cal_date enumeration)
---------------------------------------
cal_date iterates over EVERY day in the last WINDOW_DAYS calendar
days ending at REF_DATE inclusive, i.e.
    [REF_DATE - (WINDOW_DAYS - 1), REF_DATE]
This is the same window used by selection arm B, so every actionOn
captured by arm B corresponds to some cal_date in the output, and
every pending document visible on any day of that window is
reflected in the backlog counts. The sparse-output rule (rule 8)
suppresses (cal_date, branchCode) pairs that would carry zero of
all three counts.

Rules
-----
1. cal_date enumerates every day in the reporting window
   [REF_DATE - (WINDOW_DAYS - 1), REF_DATE]. Only cases that qualify
   under "Case selection" feed the aggregator; non-qualifying cases
   and their documents are dropped before bucket evaluation.
2. Bucket key for aggregation is (cal_date, branchCode). Zone,
   divisionName, and branchName are constant per branchCode and are
   carried along, not grouped on.
3. pendingDocuments - "open backlog as of cal_date". Over all documents
   of all selected cases in the branch, count documents d satisfying:
       cal_date >= d.uploadedOn
       AND ( d.status == "pending" OR cal_date < d.actionOn )
   The first clause filters out documents not yet uploaded on
   cal_date. The second clause keeps a document in the backlog if it
   is still pending (forever-pending case) or if cal_date falls before
   its action day (not yet actioned as of cal_date). A document
   actioned on cal_date itself (cal_date == d.actionOn) drops OUT of
   the backlog on that day - it is counted in processedDocuments
   instead (rule 5).
   Note: d.actionOn is the empty string when d.status == "pending";
   treat that as "no actionOn" so the comparison reduces to the
   status-pending branch.
4. submittedDocuments = COUNT of documents d satisfying:
       d.uploadedOn == cal_date
   across all selected cases in the branch. Every document thus
   contributes to exactly one cal_date bucket (its upload day).
   Document status is irrelevant - pending, accepted and rejected
   all contribute.
5. processedDocuments = COUNT of documents d satisfying:
       d.actionOn == cal_date
       AND d.status in {"accepted","rejected"}
   across all selected cases in the branch. Every processed document
   thus contributes to exactly one cal_date bucket (its action day).
6. perApprover: over the same documents that feed processedDocuments
   for this bucket, group by d.approvedBy, and for each group count
   accepted and rejected separately. Emit one element per distinct
   approvedBy, sorted by approvedBy ASC. Pending documents have no
   approver and are excluded by construction.
7. branchName, divisionName, doCode, Zone are constant per branchCode by
   construction (data_gen rule 1); take them from any record in the
   bucket.
8. Output is sorted by divisionName ASC, then branchCode ASC, then
   cal_date ASC. This keeps each branch's WINDOW_DAYS daily rows in
   chronological order within its division grouping. Buckets with
   pendingDocuments + submittedDocuments + processedDocuments = 0
   are NOT emitted (sparse output).
9. A case with zero documents does not occur in generated data
   (data_gen rule 8 guarantees >= 2); if present it contributes
   nothing to any count and can be dropped.
10. Key-name casing: output uses camelCase for every field EXCEPT "Zone",
    which matches the source record's capitalised key. "cal_date" uses
    snake_case as specified by the requirement.
11. Idempotency: for a fixed REF_DATE and a fixed source file, running
    the aggregator twice produces byte-identical output.
